2.14 自动化运维初级村-巡检-多线程

摘要

上一章节中，我们已经在执行器中集成了解析功能，可以将连接设备的执行输出解析成结构化数据；对于巡检框架来说，还缺少一部分重要的功能，那就是将解析结果进行存储、校验和导出，并且还需要通过多线程保证巡检的性能。

很多朋友在新手村的内容还没有熟悉的时候，就想学习怎么并发登陆设备下发配置，我个人非常不建议这么做；本身运维的第一原则就是稳定性，稳定性是要高于效率的，所以用多线程之前非常有必要对其进行较深的学习理解，不然盲目的并发操作设备对现网造成的风险会很大。

巡检对象

触发巡检项并且解析，然后存储，这是一个完整的动作，从面向对象的角度来看，巡检本身就是一个对象，它应该具备几个重要的属性，如下：

  • 设备相关信息，根据设备资产号即可查询到设备信息
  • 巡检项信息，根据巡检项 ID 即可查询到巡检项信息
  • 执行结果，巡检项执行后的原始输出内容
  • 解析结果，对输出进行解析的结果
  • 验证结果，针对解析后的结果进行业务逻辑上的处理
  • 时间戳，执行该巡检项的时间

还应该具备两个重要的方法，如下：

  • 创建，添加一个巡检记录
  • 执行，对某个设备执行一个或多个巡检项
  • 验证：使用代码逻辑对解析的结果做进一步处理
  • 导出，导出指定时间范围内的巡检结果

首先我们展开讲解一下为什么把这个方法设置为类方法。

类的方法分为实例方法，类方法和静态方法；

  • 实例方法很好理解，这个方法一定和该类实例化后的对象有关，所以第一个参数必须是 self，因为该方法内部会用到 self 内相关的属性或其他方法
  • 类方法的话与实例方法不同的是它和类的实例化对象无关，它内部不会用到实例相关的属性，可以理解为不需要 self 这个参数，但它第一个参数为 cls，表示类本身，可以在函数内部调用类属性和其他类方法。
  • 静态方法其实就是普通的方法，它不和这个类的产生任何交集，只是在面向对象的层面上，它属于这个类，但实际上该方法内部不会使用 self 或 cls。

根据上面的解释再来看 execute 方法，该方法首先不需要 self 参数，这时可以把它考虑声明为类方法或静态方法，但是它需要在结果处理的时候调用类中的其他验证方法对解析结果做进一步处理，所以最终选择将其声明为类方法；具体代码如下：

进一步的验证方法

导出巡检结果通常有两种场景；

  • 一种是直接将解析的结果导出。
  • 另一种是可以对解析的结果进一步处理，比如针对电源状态，可以直接输出 Normal 或 Abnormal，也可以输出成“正常”或“异常”。

这里直接用巡检项的名称作为验证方法名，并且用到 hasattr 和 getattr方法；

  • hasattr(obj, name)，obj 是一个对象类型， name 是一个字符串类型，该方法可以判断 obj 中是否有 name 名称的方法或属性，返回一个 bool 值
  • getattr(obj, name)，参数解释同上，该方法可以获取到 obj 对象中的 name 名称的方法或属性，如果没找到，会抛出异常

因为我们没有办法一次性列举所有的验证方法，但又必须能够在后续随时进行扩展，所以根据巡检项的名称来进行函数的映射是比较灵活的方式，但这需要将巡检项的名称设置为英文，如果有必要展示成中文的话也可以在数据表中单独新增一个中文字段。

巡检服务

在实现了巡检对象之后，就需要一个巡检的 service 来调用它，这个巡检的 service 就是真正实现多线程的地方。

最开始 Python 开线程或进程的方式是直接引入 threading 或 process 包，然后创建线程或进程，但这样对于并发的控制能力非常弱，很容易就造成并发太高反而降低性能的情况，因为进程和线程之间的切换也会对性能有一定的影响，所以程序开多大的并发是由操作系统的性能决定的。

如何选择多进程和多线程

多线程

IO 密集型任务选多线程。

因为IO密集型任务的主要瓶颈在于任务的 IO 阻塞，那么当某个线程遇到 IO 阻塞的时候，切换到其他的线程运行，这样就可以很大程度提高性能。

但根据描述也可以看出来，多线程其实并不是并行 （完全同时执行），而只是并发。

多进程

CPU 密集型任务选多进程

CPU 密集型任务的性能瓶颈在于对 CPU 的消耗，所以要把每个任务单独开一个进程运行到不同的 CPU 上。

多进程其实是完全意义上的并行。

开多少个进程或线程

多线程

Python 的 GIL 锁是大家一直诟病的地方，GIL 的存在会限制 Python 的多线程只能在一个 CPU 上运行；

所以用多线程处理问题时，需要通过不断调整线程数来测试多少个线程会占满 CPU；

多进程

Python 中的多进程可以很好的利用到多个 CPU，由于 CPU 密集型任务对于 CPU 的消耗很大，所以用到多进程时，进程数和 CPU 数相同。

ThreadExecutorPool

线程池控制线程的功能有很多，但最开始只需要了解几个重要的就可以了，如下：

  • ThreadPoolExecutor(max_workers=self.max_workers)，限制线程池最大线程数；
  • pool.submit(worker, arg1, arg2)，创建一个线程运行 worker 函数，并将 arg1， arg2 作为参数传递给 worker；
  • future = pool.submit(...)，创建线程后会返回一个 Future 的实例化对象，用来控制该线程；
    • 可以通过 future.attr_name = value 向 future 对象上挂载键值
    • future.result()，可以获取线程的执行结果，该值为 worker 函数的返回值
    • future.add_done_callback(callback)，可以在线程执行结束后调用自定义的 callback 函数
  • ThreadPoolExecutor 返回的线程池对象可以通过 with 上下文进行管理，会在离开 with 上下文的时候执行 pool.shutdown(wait=True)，来等待全部线程池中的线程执行结束；
